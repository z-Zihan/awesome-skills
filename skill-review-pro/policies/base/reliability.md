# base: reliability — 可靠性基础 / Reliability Foundation

所有 Skill 类型共享的可靠性评审基础。

## 核心问题 / Core Question

这个 Skill 能不能稳定、正确地完成任务？

## 二级观察项 / Secondary Observations

- **Positioning Clarity / 定位清晰度** — 能否快速理解 Skill 干什么、不做什么
- **Instruction Clarity / 指令明确性** — 指令有无歧义、矛盾、缺失
- **Boundary Rationality / 边界合理性** — 职责是否聚焦，有无膨胀
- **Actionability / 可执行性** — AI 能否据此行动（区分 Skill 和知识文章）
- **Adversarial Robustness / 对抗鲁棒性** — 模糊/越界/矛盾输入时是否优雅处理
- **Degradation Strategy / 降级策略** — 依赖不可用或异常时有无 fallback

## 评审要点 / Review Points

### 必查项 / Must-Check

- 5 秒内能否理解 Skill 的输入/输出/边界
- 核心流程是否有明确的步骤说明
- 是否有"不做什么"的边界定义
- 指令是否存在矛盾（如"严格按模板"和"自由发挥"并存）
- 模糊指令（如"适当处理"、"合理调整"）是否给出了判断标准
- **Skill vs 知识文章**：是否存在"knowledge-not-skill"问题——内容是知识/教程/参考，但无可执行指令？

### 对抗检查清单 / Adversarial Checklist

> 从 37 个真实评审数据归纳的高频问题。评审时逐一快速检查。

| # | 对抗场景 | 期望行为 | 常见失败模式 |
|---|---------|---------|-------------|
| A1 | 模糊输入："帮我搞一下" | 应澄清需求，不应胡乱执行 | 无需求澄清流程，直接执行 |
| A2 | 越界请求：请求不在 Skill 职责范围内 | 应拒绝或重定向 | 无边界定义，强行处理 |
| A3 | 矛盾请求：同时要求冲突的目标 | 应指出矛盾并请求澄清 | 盲目执行其中一个 |
| A4 | 依赖不可用：引用的工具/文件不存在 | 应有降级或明确报错 | 静默失败或崩溃 |
| A5 | 硬编码路径/ID：环境不同时 | 应可配置或参数化 | 路径/ID 写死，迁移后崩溃 |

检查方式：对照上述场景，判断 Skill 的指令是否覆盖了对应的处理规则。未覆盖 = 扣分。不需要实际执行测试。

## 评分锚点 / Scoring Anchors (Reliability 维度内)

- 定位清晰：能理解且边界明确
- 指令明确：无歧义可执行，模糊指令有判断标准
- 可执行性：有具体步骤，AI 能据此行动（非纯知识）
- 对抗鲁棒：模糊/越界/矛盾/异常场景有明确处理规则
- 降级策略：依赖不可用时有 fallback

综合锚点（满分 = 上述全部满足）：
- 满分 = 全部满足，包括对抗检查清单 5/5
- 70% = 有边界和可执行性，但缺 2-3 项对抗覆盖
- 40% = 有基本流程但缺边界或降级策略
- 15% = 不是真正的 Skill（knowledge-not-skill）
