## 推荐技术栈（版本锚定）

| 类别 | 推荐技术 | 版本要求 | 说明 |
|---|---|---|---|
| **关系型数据库** | PostgreSQL | ≥ 16（使用最新稳定版） | 功能最全的开源关系型数据库 |
| **关系型数据库（备选）** | MySQL | ≥ 8.0（使用最新稳定版） | 国内生态更成熟 |
| **文档型数据库** | MongoDB | ≥ 7.0（使用最新稳定版） | 灵活 schema，适合快速迭代 |
| **缓存** | Redis | ≥ 7.2（使用最新稳定版） | 内存数据库，缓存/会话/消息队列 |
| **ORM（TS）** | Prisma | ≥ 5.x（使用最新稳定版） | 类型安全，自动迁移 |
| **ORM（Go）** | GORM | ≥ 1.25（使用最新稳定版） | Go 生态主流 |
| **ORM（Python）** | SQLAlchemy | ≥ 2.x（使用最新稳定版） | Python 生态成熟方案 |
| **数据库迁移工具** | 根据语言选择 | 最新稳定版 | Prisma Migrate / Alembic / golang-migrate |
| **数据库客户端** | DBeaver / DataGrip | 最新稳定版 | 通用数据库管理 GUI |

## Phase 2 数据库分支流程

### 步骤 2.1 — 数据库基础概念

1. **什么是数据库，为什么需要它**
   - 用"文件柜"类比——"数据库就是一个有组织的文件柜，可以快速查找、安全存储、多人共享"
   - 不用数据库的问题——"数据散落在各处，无法高效查询，容易丢失和不一致"

2. **关系型 vs 非关系型**
   - 关系型（PostgreSQL/MySQL）——"像 Excel 表格，有严格的列定义，表之间有关联"
   - 非关系型（MongoDB）——"像文件夹，每份文档可以有不同的结构，灵活但需要自律"
   - 选择建议——"90% 场景首选关系型；需要灵活 schema 或文档结构用 MongoDB"

3. **SQL 基础语法**
   - SELECT / INSERT / UPDATE / DELETE——"增删改查，数据库的四大基本操作"
   - WHERE 条件过滤
   - ORDER BY 排序
   - LIMIT 分页

⏸ **暂停点**：确认用户理解关系型/非关系型的区别，能写出基本 SQL 后再继续。如果用户已有 SQL 基础，快速跳过。

---

### 步骤 2.2 — 数据库设计与建模

1. **表设计基础**
   - 主键（Primary Key）——"每行数据的唯一标识，像身份证号"
   - 外键（Foreign Key）——"表之间的关联关系，像家庭成员关系"
   - 索引（Index）——"数据库的目录，加速查询但占空间"
   - 什么时候该加索引，什么时候不该

2. **ER 图与关系设计**
   - 一对一、一对多、多对多关系
   - 多对多关系的中间表设计
   - 范式与反范式——"范式减少冗余，反范式提升查询性能，实际项目常在两者间平衡"

3. **用用户熟悉的领域举例**
   - 用户-文章-评论系统
   - 订单-商品-用户系统
   - 根据用户项目需求定制

4. **数据类型选择**
   - 整数、浮点、字符串、日期、JSON
   - PostgreSQL 的特色类型（JSONB、ARRAY、UUID）

⏸ **暂停点**：确认用户完成表结构设计后再继续。让用户画出 ER 图或写出建表 SQL。

---

### 步骤 2.3 — 查询进阶

1. **JOIN 多表查询**
   - INNER JOIN / LEFT JOIN / RIGHT JOIN / FULL JOIN
   - 用"拼图"类比——"JOIN 就是把分散在不同表里的数据，按关联条件拼在一起"
   - 常见陷阱：笛卡尔积——"忘记加 ON 条件会把每行和每行组合，数据爆炸"

2. **聚合查询**
   - GROUP BY / HAVING
   - COUNT / SUM / AVG / MAX / MIN
   - 用"分组统计"类比——"GROUP BY 就像 Excel 的分类汇总"

3. **子查询与窗口函数**
   - 子查询——"查询里面套查询，先算内层再算外层"
   - 窗口函数（ROW_NUMBER / RANK / LAG / LEAD）——"在不改变行数的情况下做分组计算，排行榜常用"
   - CTE（WITH 子句）——"给子查询起个名字，代码更清晰"

⏸ **暂停点**：确认用户能写出复杂查询后再继续。给出练习题让用户动手写。

---

### 步骤 2.4 — 性能优化

1. **索引设计与优化策略**
   - 什么字段该加索引——"WHERE、JOIN、ORDER BY 涉及的列"
   - 复合索引与最左前缀原则——"索引 (a, b, c) 只能用于 a、(a,b)、(a,b,c) 的查询"
   - 索引不是越多越好——"每个索引都有写入开销，索引过多会拖慢 INSERT/UPDATE"

2. **慢查询排查**
   - EXPLAIN / EXPLAIN ANALYZE——"查看查询执行计划，就像看导航路线"
   - 全表扫描 vs 索引扫描——"全表扫描就是挨个翻书找，索引扫描就是先查目录再翻"
   - 常见慢查询模式及优化方法

3. **连接池配置**
   - 为什么需要连接池——"建立数据库连接很贵，连接池复用连接避免反复创建"
   - 推荐配置——"开发环境 5-10 连接，生产环境根据并发量调整"
   - 连接池监控

⏸ **暂停点**：每完成一个优化主题后暂停。性能优化是实践性很强的环节，建议用真实数据练习。

## 理想结果

学完数据库方向后，用户应该能够：

- ✅ 理解关系型和非关系型数据库的核心概念
- ✅ 设计合理的数据库表结构和关系
- ✅ 熟练编写 SQL 查询（包括 JOIN、聚合、窗口函数）
- ✅ 分析和优化慢查询
- ✅ 配置数据库连接池
- ✅ 完成数据库从设计到生产配置的完整流程
