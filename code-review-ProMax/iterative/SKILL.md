## 迭代 Review（修复后二次/多次 Review）

用户提交修复后再次 review 时（"改好了，再看一下"），采用**精简模式**。

### 核心原则

- **始终审查最新代码**，不是和旧版本对比。重新审查相关文件当前状态
- 未修复/部分修复/改错的问题，**必须重新给出具体修复建议**，不只标记状态
- **核心判断：修复是否解决根因，还是只是 suppress 表象**

### 精简报告规则

- 已确认符合预期 → 不提
- 已修复 → ✅ 标记，不展开
- ❌ 未修复 → 重新分析当前代码，给出具体修复建议
- ⚠️ 部分修复 → 说明还差什么 + 剩余部分修复建议
- 修复改错 → ❌ 标记，分析为什么改错 + 正确方向
- 新引入问题 → 正常详细输出
- Suppress 检测 → 删除检查/忽略错误/try-catch吞异常 → 🚫 "疑似 suppress"，要求确认
- 关联遗漏 → 修了 A 文件 bug，检查 B 文件是否同模式需同步修

### 报告模板

```markdown
## 二次 Review

**结论**: 可直接合入 / 仍有问题需修复

### 修复确认

| # | 原问题 | 状态 |
|---|--------|------|
| 1 | 简要描述 | ✅ 已修复 |
| 2 | 简要描述 | ❌ 未修复 |
| 3 | 简要描述 | ⚠️ 部分修复（还差 XXX） |

> 全部 ✅ 且无新问题 → "可直接合入"，结束。

### 未修复/修复有误的问题

| # | 原问题 | 当前状态 | 修复建议 |
|---|--------|----------|----------|

> 全部已修复则写"无"。

### 新增问题

| # | 严重度 | 位置 | 问题描述 | 修复建议 |
|---|--------|------|----------|----------|

> 无则写"无新增问题"。

### Suppress 检测与关联遗漏

| # | 修复方式 | 检测结果 |
|---|----------|----------|

> 无则写"无"。
```

### 多轮迭代

- 继续精简模式，每轮审查最新代码
- 只关注：上轮遗留 + 本轮新增
- 持续有未修复问题 → 持续给具体修复方向，不建议"接受现状"
- 结论"可直接合入"时才省略修复指令

### 问题展开分析

触发："展开第 N 个"、"详细分析一下"。

输出：复现场景 → 根因 → 影响范围 → 实际后果(具体场景) → 修复思路(含备选方案对比) → 回归风险

- 只展开用户要求的问题，不顺带分析其他
- 后果描述要具体有场景，不写"可能产生不可预期的问题"

> 问题格式同主 Review 报告 §3，不再重复定义。
